test: use actually-distinct cells in cross-cell recompute test

The previous version of "is recomputed when the user visits a
new cell" used coords (22.298, 114.170) and (22.315, 114.170) —
both in Yau Tsim Mong. Empirically these hash to the SAME
geohash-5 cell (wkfg8), because geohash-5 cells are ~20 km tall
in the latitude direction (see lib/utils/geohash.dart for the
size table). The 1.9 km shift between the two fixes stays well
inside one cell.

That meant the test wasn't actually exercising the cross-cell
recompute path. It would have passed even if _recomputeSuggestion
were never called on cell change (verified by temporarily
commenting out the _recomputeSuggestion() call — the test
passed).

Fix: pick coords that hash to DIFFERENT geohash-5 cells in the
SAME district (so the test focuses on cell-crossing, not
district-crossing). New coords are (22.270, 114.140) and
(22.270, 114.170), both in Central and Western, hashing to
wkfft and wkfg8 respectively.

Verified the test now actually exercises the contract:
- With _recomputeSuggestion() called: test passes
- With _recomputeSuggestion() commented out: test fails
  (suggestion stays null because the recompute never fires)
